APPENDIX — CONTRACTOR DIAGNOSTIC EXPERIMENT (off-plan, Bret-approved): can Phase 2
run FULLY LOCAL with no cloud key?  ->  NO (local server 400s the ADK request)
-------------------------------------------------------------------------------
NOT part of Jonah's plan; a self-initiated in-scope experiment. Overlay created and
deleted; clone left clean.

Setup: created a gitignored contract.local.yaml next to
src/nodes/node_agent_orchestrator/contract.yaml with
  config.agent_runtime.adk_model_ref: local_deepseek_reasoning
(the loader supports this overlay seam; contract.py:103). This repoints the agent's
ADK reasoning backend from the default cloud_gemini to the local vLLM tier at :8001.

What worked (no cloud key needed):
  - overlay loaded, deep-merged over the default contract;
  - ADK selected the local OpenAI-compatible adapter (_OpenAICompatibleLlm,
    Authorization: Bearer EMPTY);
  - endpoint rewrite via SEA_LOCAL_INFERENCE_BASE_URL resolved to
    http://100.109.203.94:8001/v1/chat/completions (the same proven endpoint);
  - got PAST the cloud_gemini key stop entirely.

What failed:
  - httpx.HTTPStatusError: 400 Bad Request from the local endpoint, raised inside
    google/adk/models/apigee_llm.py during the agent loop ->
    "RuntimeError: ADK runner background thread failed".

Root cause ISOLATED (captured the exact request + 400 body via httpx instrumentation,
2026-06-08T18:3xZ; full capture in local-adk-400-capture.json):

  400 response body from the local llama.cpp server (:8001):
    "Unable to generate parser for this template. Automatic parser generation
     failed: JSON schema conversion failed: Unrecognized schema: {\"type\":\"STRING\"}"
     (repeated for each tool parameter)

  The request the ADK adapter sent serializes every tool-function parameter with
  GEMINI-DIALECT JSON Schema types — UPPERCASE — e.g.
    "parameters":{"properties":{"description":{"type":"STRING"}},"type":"OBJECT"}
  Gemini's function-calling schema uses STRING/OBJECT/INTEGER. The local
  OpenAI-compatible server (llama.cpp build b9206) requires standard lowercase
  JSON Schema ("string"/"object") and rejects the uppercase types when building its
  tool-calling grammar -> 400.

  Why earlier probe got 200: my direct probe used a hand-written lowercase tool
  ({"type":"object"/"string"}), which the server accepts. The agent's REAL tools come
  from ADK FunctionDeclaration objects (Gemini Schema), serialized uppercase. So it
  is specifically the Gemini->OpenAI tool-schema dialect that is not converted.

  Where: SEA's _OpenAICompatibleLlm adapter / build_http_chat_completions_model in
  src/nodes/node_agent_orchestrator/adk_backend_adapters.py passes the request
  through Google ADK's CompletionsHTTPClient (apigee_llm) WITHOUT normalizing the
  function-declaration schema types to lowercase for the OpenAI-compatible target.
  The Gemini backend works because Gemini accepts uppercase natively.

  Likely fix: when targeting an OpenAI-compatible endpoint, lowercase the JSON Schema
  "type" values (and any other Gemini->OpenAI schema normalization) on the tool
  function parameters before sending. Candidate site: the _OpenAICompatibleLlm
  adapter path. (Could not isolate without authorization to dig; observation only.)

Conclusion: fully-local SEA --agent generation fails because the local-ADK adapter
ships Gemini-dialect (uppercase) tool schemas to the OpenAI-compatible local server,
which rejects them (400). The local INFERENCE path is proven (Phase 3). The local
ORCHESTRATION path (ADK agent loop on a local model) is blocked on this single schema
conversion. Cloud Gemini remains the only orchestrator backend that completes today,
and that is out of today's scope. Not a demo blocker; a gap in the advertised
local-ADK fallback.
